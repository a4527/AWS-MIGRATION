# 00. Requirements

## 프로젝트 개요

기업 내부 파일 공유 서비스를 온프레미스 환경에서 먼저 구현하고, 동일한 기능을 AWS 클라우드 네이티브 구조로 이전한다.

## 핵심 범위

### 포함 범위

- 사용자 인증 및 인가
- 파일 업로드, 다운로드, 삭제
- 파일 메타데이터 관리
- 캐시 기반 성능 개선
- 객체 스토리지 사용
- 컨테이너 기반 실행 환경
- AWS 마이그레이션 및 운영 자동화

### 제외 또는 후순위 범위

- 파일 버전 관리
- 공개 공유 링크 생성
- 대용량 분할 업로드
- 파일 미리보기
- 고급 검색/필터링
- 관리자 대시보드

### 설계 원칙

- 온프레미스와 AWS 구현은 같은 기능 기준으로 비교 가능해야 한다.
- 기능 범위는 단순하게 유지하고, 저장소와 배포 환경 변경에 집중한다.
- 파일 바이너리와 메타데이터는 분리해서 관리한다.
- Day 1 문서에서는 구현 방식보다 기능 경계와 책임을 먼저 확정한다.

## 기능 요구사항

### 인증

- JWT 기반 로그인/인가
- 사용자별 접근 권한 분리
- 인증 실패 시 명확한 에러 응답

### 파일 관리

- 파일 업로드
- 파일 다운로드
- 파일 삭제
- 파일 메타데이터 조회
- 파일 크기 및 확장자 검증

### 사용자 관리

- 사용자 등록
- 로그인
- 사용자 정보 조회
- 권한에 따른 접근 제한

### 운영 기능

- Redis 캐싱
- 업로드 이력 및 기본 감사 로그
- 에러 추적과 장애 대응 문서화
- S3 이벤트 기반 서버리스 후처리

### 서버리스 처리

- 파일 업로드 이후 필요한 후처리 작업은 AWS Lambda로 분리한다.
- 후처리 내용은 바이러스 검사와 메타데이터 추출이다.
- 서버리스 작업은 S3 이벤트를 트리거로 실행한다.
- 서버리스 처리 결과는 CloudWatch 로그와 운영 이력으로 남긴다.
- 온프레미스 환경에서는 해당 작업을 동기 처리 또는 후속 배치로 대체한다.

## 사용자 역할

### 일반 사용자

- 회원가입과 로그인을 수행한다.
- 본인이 업로드한 파일을 조회, 다운로드, 삭제할 수 있다.
- 권한이 부여된 파일에 한해 접근할 수 있다.

### 관리자

- 사용자 정보와 파일 메타데이터를 조회할 수 있다.
- 운영 관점에서 업로드 이력과 장애 상황을 확인할 수 있다.
- 필요 시 권한 정책과 보안 설정을 점검한다.

## 파일 라이프사이클

### 상태 흐름

- `uploaded`: 파일이 업로드되어 메타데이터가 등록된 상태
- `scanning`: 바이러스 검사와 메타데이터 추출이 진행 중인 상태
- `available`: 조회와 다운로드가 가능한 상태
- `quarantined`: 검사 결과 이상이 발견되어 접근이 차단된 상태
- `deleted`: 삭제가 완료된 상태

### 상태 전이 원칙

- 업로드 성공 시 `uploaded` 상태로 시작한다.
- 서버리스 후처리 또는 동기 후처리가 완료되면 `available`로 전환한다.
- 검사 중에는 `scanning` 상태를 유지한다.
- 바이러스가 검출되거나 후처리 실패가 치명적이면 `quarantined` 상태로 전환한다.
- 삭제 요청이 완료되면 `deleted` 상태로 전환한다.
- `deleted` 상태의 파일은 일반 조회와 다운로드 대상에서 제외한다.

### 삭제 정책

- 기본 정책은 논리 삭제를 우선한다.
- 실제 바이너리 정리는 별도 운영 절차나 배치 작업으로 분리할 수 있다.
- 메타데이터와 실제 파일 객체의 제거 시점을 분리해 장애 복구 가능성을 확보한다.

### 실패 처리

- 업로드 도중 실패하면 메타데이터와 저장소 객체를 함께 정리한다.
- 다운로드 권한이 없으면 파일 정보는 노출하되 실제 바이너리는 제공하지 않는다.
- `quarantined` 상태의 파일은 다운로드 대상에서 제외한다.
- 존재하지 않는 파일은 명확한 에러 코드로 응답한다.

## 비기능 요구사항

- 보안: JWT, HTTPS, IAM, 비밀정보 분리
- 가용성: Docker Compose와 ECS에서 안정적 구동
- 확장성: S3, Aurora, ElastiCache 기반 수평 확장
- 유지보수성: 문서 중심 개발
- 관측성: CloudWatch 및 애플리케이션 로그

## 성공 기준

- 온프레미스 버전이 Docker Compose로 정상 동작한다.
- AWS 버전이 ECS와 관리형 서비스로 정상 동작한다.
- 마이그레이션 전후 기능이 동등하게 유지된다.
- 문서만 보고도 재현 가능한 수준의 설명이 제공된다.

## Day 1 기준

- 서비스 범위와 비범위가 문서로 고정된다.
- 사용자 역할이 최소 단위로 정의된다.
- 파일 상태와 삭제 정책이 고정된다.
- 이후 API, DB, 아키텍처 문서가 동일한 용어를 사용한다.
